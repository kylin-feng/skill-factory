# Skill工厂 · 从需求到可运行Skill

> 「工具的价值，在于让能力可复制。」

## 核心理念

Skill工厂不是写提示词，是**构建可复用的认知工具**。

一个好的 Skill 具备：
- **明确的触发条件**：什么时候激活，比什么都说得多
- **清晰的执行流程**：遇到什么做什么，有检查点
- **诚实的边界**：做不到的事情提前说清
- **可验证的输出**：用户能判断结果是否正确

**六大 Skill 类型**：

| 类型 | 核心价值 | 典型案例 |
|------|---------|---------|
| **人物视角** | 用某人的思维框架分析问题 | 费曼视角、芒格视角 |
| **工作流** | 自动化多步骤重复操作 | PR Review、Deploy流程 |
| **领域专家** | 深度领域知识的交互顾问 | SQL优化专家、安全审计 |
| **代码规范** | 检查并强制执行代码标准 | 团队规范卫士、架构守护者 |
| **内容创作** | 特定格式/风格的内容生成 | 周报生成、技术文档 |
| **工具集成** | 连接外部服务的操作桥梁 | GitHub助手、Linear工单 |

## 执行流程

### Phase 0: 需求接收与类型识别

收到用户输入后，分析属于哪种路径：

#### 路径A：用户明确指定类型
直接进入 Phase 0.5 确认细节。

#### 路径B：用户描述需求但未指定类型
运用类型识别规则：

**类型识别规则**（按关键词和意图匹配）：

| 用户说的 | 识别为 | 识别依据 |
|---------|--------|---------|
| 「用XX的角度看问题」「XX会怎么想」 | 人物视角 | 模拟特定人的思维方式 |
| 「每次XX都要XX」「自动化XX流程」 | 工作流 | 重复性多步骤操作 |
| 「我需要一个XX专家」「帮我分析XX」 | 领域专家 | 深度知识的持续咨询 |
| 「检查代码是否符合XX」「强制执行XX规范」 | 代码规范 | 代码质量/风格约束 |
| 「帮我写XX格式的YY」「生成XX模板」 | 内容创作 | 特定格式的内容产出 |
| 「连接/集成XX工具」「操作XX平台」 | 工具集成 | 与外部服务交互 |

**无法识别时**：向用户展示6种类型简介，让其选择。

---

### Phase 0.5: 需求澄清对话

根据识别的类型，针对性追问（每种类型最多问3个问题）：

#### 人物视角
1. 这个人是谁？有没有聚焦的领域？
2. 主要用途：思维顾问 / 决策参考 / 角色扮演？
3. 有没有本地素材（书籍/演讲/访谈）？

#### 工作流
1. 这个流程的起点是什么？（触发条件）
2. 主要步骤有哪些？（可以不完整，后续补充）
3. 最终产出是什么？（文件/报告/操作/消息）

#### 领域专家
1. 具体是哪个领域/技术栈？
2. 主要解决什么类型的问题？
3. 专家的交互模式：问答 / 审查 / 建议 / 生成？

#### 代码规范
1. 规范适用于哪个项目/语言/框架？
2. 规范的来源：团队文档 / 官方标准 / 自定义？
3. 检查时机：提交前 / 代码审查 / 实时提醒？

#### 内容创作
1. 内容的类型和格式？（周报/文档/邮件）
2. 目标受众是谁？
3. 有没有现有的模板或示例？

#### 工具集成
1. 要集成哪个工具？（GitHub/Jira/Slack等）
2. 主要执行什么操作？
3. 触发方式：手动调用 / 特定事件 / 定时？

**用户给的信息足够时，直接推进，不要问不必要的问题。**

### Phase 1: 素材收集（按类型）

**人物视角** → 多Agent并行调研（参考女娲模式）：
- 著作与系统思考
- 长对话与即兴表达
- 表达风格与DNA
- 外部评价与批评
- 决策记录

**工作流** → 分析现有流程：
- 读取相关配置文件、脚本、文档
- 识别现有步骤的工具和命令
- 标注可以自动化 vs 需要人工确认的环节

**领域专家** → 构建知识体系：
- WebSearch 收集该领域的核心知识框架
- 识别常见问题模式和反模式
- 收集权威参考资料和工具

**代码规范** → 提取规范内容：
- 读取项目现有配置（.eslintrc / .prettierrc / pyproject.toml等）
- 读取团队文档（如用户提供）
- 提炼核查清单

**内容创作** → 理解格式要求：
- 分析用户提供的示例
- 提取结构模式、语气风格、关键要素
- 建立生成规则

**工具集成** → 了解工具能力：
- 确认工具的认证方式（API key / OAuth）
- 梳理所需操作的API或命令
- 标注权限要求和安全注意事项

### Phase 2: Skill 构建

读取对应类型的模板文件（`templates/[type].md`），填充内容。

**通用必填字段**（所有类型）：
- `name`：小写横线连接，描述性命名
- `description`：触发词 + 用途 + 场景（这是最重要的字段）
- 执行流程：至少3个步骤，带检查点
- 诚实边界：做不到的事情

**类型专属字段**：

| 类型 | 专属必填 |
|------|---------|
| 人物视角 | 心智模型(3-7个)、表达DNA、人物时间线 |
| 工作流 | 触发条件、步骤序列、错误处理、产出格式 |
| 领域专家 | 知识框架、问题分类、决策树、参考资源 |
| 代码规范 | 规则清单、示例（正确/错误）、自动修复指引 |
| 内容创作 | 结构模板、风格规则、变量列表、示例输出 |
| 工具集成 | 认证方式、操作命令、权限说明、错误处理 |

#### 关键：Description 的写法

description 是 Skill 被触发的关键。写法要覆盖：
1. **明确触发词**（用户会说的完整短语）
2. **模糊触发词**（用户描述需求但没说 skill 名）
3. **使用场景**（这个 skill 在什么情况下有用）
4. **不触发的情况**（避免误激活）

---

### Phase 3: 验证与测试

每个 Skill 生成后，执行3项验证：

#### 3.1 触发验证
准备5个用户可能说的句子，检查 description 是否能覆盖：
- 2个明确触发（直接说 skill 名）
- 2个模糊触发（描述需求）
- 1个边缘情况（模糊地带）

#### 3.2 执行验证
给 Skill 一个典型输入，干跑执行流程：
- 步骤是否清晰？
- 检查点是否合理？
- 产出格式是否明确？

#### 3.3 边界验证
给一个超出 Skill 能力范围的输入，检查：
- 是否有明确的边界说明？
- 拒绝时的提示是否有帮助？

### Phase 4: 安装与交付

**生成的文件结构**：
```
.claude/skills/[skill-name]/
├── SKILL.md          # 主 Skill 文件（核心交付物）
└── references/       # 辅助参考（如有需要）
    ├── rules.md      # 详细规则（代码规范类）
    ├── template.md   # 内容模板（内容创作类）
    └── research/     # 调研资料（人物视角类）
```

**安装确认**：
完成后告知用户：
1. Skill 已生成至 `.claude/skills/[skill-name]/SKILL.md`
2. 触发词列表
3. 一个立即可以测试的示例输入

## 品味守则

**什么是好的 Skill**：
- 用户 3 秒内知道这个 Skill 是干什么的
- 执行流程有可预测的产出
- 边界清晰，不过度承诺

**什么是烂的 Skill**：
- description 太泛，什么都触发（或什么都不触发）
- 执行流程是「帮用户做任何事」
- 没有检查点，一把梭到底

**绝不做的事**：
- 为了「看起来厉害」而增加不必要的步骤
- 在没有足够信息时强行生成
- 把通用 AI 能力包装成 Skill（那不是 Skill，只是提示词）

## 特殊场景

### Skill 组合（Composition）
如果用户的需求跨越多个类型，可以构建「组合 Skill」：
- 主 Skill 负责流程编排
- 调用子 Skill 处理专项任务
- 在 description 中说明依赖关系

### 从现有 Skill 升级
当用户说「升级我的 XX skill」时：
1. 读取现有 SKILL.md
2. 对比六大类型的标准，找出薄弱项
3. 增量补充，不推倒重来

### 从 nuwa-skill 迁移
如果用户有 nuwa-skill 生成的人物视角 Skill，可以：
1. 按本工厂的人物视角模板评审
2. 补充 Agentic Protocol（回答工作流）
3. 补充 Skill 组合能力

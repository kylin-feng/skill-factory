# 工作周报生成器

> 周报的价值不是汇报，是让你自己看清楚这周做了什么。

## 触发后立即执行

**Step 1: 信息收集**

优先顺序：
1. **自动获取 git 记录**（最客观）：
   ```bash
   git log --since="7 days ago" --author="$(git config user.name)" \
     --pretty=format:"%ad %s" --date=short --no-merges 2>/dev/null
   ```
   如果在 git 仓库内且有本周提交，展示记录，询问是否补充其他信息。

2. **用户粘贴工作笔记/流水账**：
   直接分析整理，不追问。

3. **什么都没有**：
   问用户：「简单说说这周做了什么？不用完整，关键词也行」

**Step 2: 补充信息收集**（快速，最多2问）

- 「这周有没有遇到什么卡点或解决的难题？」（填「本周挑战」部分）
- 「下周主要打算做什么？」（填「下周计划」部分）

用户说「没有」或「不用」→ 跳过，对应部分填「暂无」。

## 周报生成规则

### 内容整理规则

**对 git commits 的处理**：
- 合并同一功能的多个小提交
- 翻译技术性描述为可读语言：`fix: null pointer in UserService` → 「修复用户服务崩溃问题」
- 去掉无意义提交：merge commits、typo fixes
- 按功能模块分组，不按提交时间排列

**对流水账的处理**：
- 提炼关键成果，去掉过程细节
- 量化可以量化的：「开了很多会」→「参与项目讨论会议 N 次」
- 区分「做了什么」和「正在做什么」（后者归入下周计划）

**写作原则**：
- 动词开头：完成、推进、优化、解决、参与、调研
- 有结果的说结果：「优化了查询」→「优化用户列表查询，响应时间从 2s 降至 300ms」
- 没有量化数据时，描述清楚就够，不要编造数字

### 标准输出格式

```markdown
# 工作周报 · [年]-W[周数]（[月/日] - [月/日]）

## 本周完成

### [模块/项目名1]
- [成果1，动词开头，有具体产出]
- [成果2]

### [模块/项目名2]
- [成果1]

## 本周挑战
- [遇到的问题 + 解决方式]（如无填「无」）

## 下周计划
- [ ] [任务1]
- [ ] [任务2]
- [ ] [任务3]

---
*周报由 git-workflow skill 辅助生成*
```

## 风格变体

用户可以指定风格：

**简洁版**（给不爱写周报的）：
```markdown
## W[N] 周报

**做了**：[3-5个要点，一句话一条]
**卡点**：[1-2条或无]
**下周**：[2-3个计划]
```

**详细版**（给需要展示工作量的）：
在标准格式基础上，每个成果补充：
- 技术方案简述
- 影响范围或相关人员
- 后续跟进事项

用户没有指定 → 使用标准格式。

## 二次修改

用户说「改一下」时，先确认修改方向：
- 「更简洁」→ 压缩到简洁版
- 「更专业」→ 调整措辞，增加量化描述
- 「某部分改改」→ 只改指定部分
- 「换个格式」→ 询问目标格式

## 诚实边界

- 从 git 记录推断的工作描述可能不准确，请核对后再发送
- 无法自动感知「做了但没提交」的工作（如会议、调研、文档）
- 不了解你的团队用语和项目背景，可能需要你微调措辞

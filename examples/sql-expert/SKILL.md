## 知识框架

### 索引策略（最高频问题）

**何时建索引**：
- WHERE 子句中的过滤条件
- JOIN ON 条件的关联字段
- ORDER BY / GROUP BY 的字段
- 高选择性字段（区分度高）

**何时不建索引**：
- 小表（全表扫描更快）
- 频繁写入、极少查询的字段
- 低选择性字段（如性别、状态只有几个值）

**索引类型选择**：
- B树索引：适合范围查询、等值查询（默认类型，用于90%场景）
- Hash索引：只适合等值查询（MySQL MyISAM / Memory）
- 全文索引：文本搜索（别用 LIKE '%keyword%'）
- 复合索引：遵循最左前缀原则

### 常见性能杀手

| 反模式 | 原因 | 修复 |
|--------|------|------|
| `SELECT *` | 传输不必要的数据 | 只选需要的列 |
| `LIKE '%keyword%'` | 无法使用索引 | 全文索引或搜索引擎 |
| `WHERE YEAR(create_time) = 2024` | 函数破坏索引 | `WHERE create_time BETWEEN ...` |
| `NOT IN (子查询)` | 子查询结果含NULL时逻辑错误 | 用 NOT EXISTS 或 LEFT JOIN |
| 大偏移量分页 `LIMIT 100000, 10` | 扫描大量行 | 游标分页（WHERE id > last_id） |
| N+1 查询 | 循环中执行查询 | JOIN 或 IN 一次查出 |

### 执行计划解读速查

```
type 列（越靠前越好）：
system > const > eq_ref > ref > range > index > ALL

关键警告：
- type = ALL：全表扫描，通常需要加索引
- Extra = Using filesort：额外排序，考虑加排序字段索引
- Extra = Using temporary：临时表，GROUP BY/ORDER BY 字段不同
- rows 很大：扫描行数多，需要优化
```

## 标准问答

**Q: 什么时候用子查询，什么时候用JOIN？**
大多数情况下 JOIN 性能更好，因为优化器对 JOIN 的优化更成熟。
子查询适合：语义上确实是「先查出结果再用」的逻辑，或者子查询只执行一次的关联子查询。
避免：在 WHERE 中用依赖外部查询的相关子查询（用 JOIN 替代）。

**Q: 为什么加了索引还是慢？**
常见原因：
1. 索引选择性太低（如只有True/False的字段）
2. SQL写法导致索引失效（如对字段用了函数）
3. 数据量太大，即使用了索引也扫描很多行
4. 索引统计信息过期（执行 ANALYZE TABLE）
5. 实际上没有用到你以为用到的索引（看 EXPLAIN）
